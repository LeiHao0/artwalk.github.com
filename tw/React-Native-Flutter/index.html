作為一個愉快地遵循  步驟的 iOS 開發人員，當我涉足 iOS 開發人員之初，便使用了 Swift，Storyboard 和 AutoLayout。

我認為這就是為什麼大多數本機開發人員甚至都不願意學習 React Native 的原因。

兩年前，我建立了 Mattermost 並在以前的公司中進行了一些更改以取代 Slack，這是我接觸的第一個 React Native 項目。

不幸的是，丟失代碼非常容易，而且每次我想更改某些內容時，我都必須搜索整個項目，沒有 Storyboard 讓我知道誰告訴誰。

我和我以前的同事聊天，後者是高級前端。他幫助我啟動了更新服務器，該服務器也是用 Node.js 編寫的。我離開時將這個項目移交給他。

據他介紹，這是因為他不知道如何編寫本機代碼，就像 React React 類似於 React.js 一樣，為什麼不使用它同時嘗試 iOS 和 Android。然後我問他是否知道如何快速發展，他會喜歡哪種技術。您可以猜出答案 😄。

因為從我的角度來看，有時即使在快速代碼中也難以實現功能，這絕對是另一項技術要實現的漫長的路要走。

然後，這幾天有機會進行一個演示，其中包含嵌套的磁鐵滾動視圖，按鈕和導航。我不再使用本機代碼，而是走出舒適區，嘗試使用 React Native 和 Flutter。

  • 觀看次數
    • ListView，ScrollView，標籤，文本，TextField
  • 動作
    • 拖動，單擊，滾動
    • 回調阻止或委託查看事件

因此，如果您精通其中任何一種，剩下的就是 Google 如何在該框架中實現它或閱讀其手冊。

反應本機

React Native 令人印象深刻的是 state

它可以通過 Swift 中的變量的didSet來實現，還有很多框架可以做到，例如 RxSwift，PromiseKit

撲

我曾經很好奇某些應用程序 UX 非常像由本機開發的，但以某種方式感到怪異。現在我知道它在顫抖。

  • 這是我第一次編寫一次並獲得兩個應用程序(使用 Android 設置 React Native 時遇到了很多麻煩)
  • 0.9 在熱裝上有錯誤
  • 在 Visual Studio Code 上的開發相當不錯，這讓我回想起 Golang 的過去
  • 醜陋的代碼風格(他們對此很認真嗎？這是一個官方示例)

無論如何，我真的給它留下了深刻的印象，它跳過了本機組件，並編寫了自己的 2D 渲染引擎。比 React Native 天才得多。

因此，就我個人而言，我認為 Flutter 比 React Native 更具潛力。以及有許多公司回滾到本地。
