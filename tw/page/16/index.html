NTFS 磁盤文件的快速定位

摘要

本文介紹了在windows NTFS 磁盤格式下，枚舉硬盤上所有文件以及文件夾的名稱，以及利用C++的STL 構建哈希表，還有MFC 的GUI 與worker 線程，最終根據用戶輸入的關鍵字，實現像google 關鍵字查找那樣簡單的搜索，然後瞬間返回所有匹配的文件/文件夾以及遞歸得到該文件/文件夾的系統路徑。

關鍵詞：NTFS, 快速，關鍵詞，查找，文件路徑

1 引言

Windows 的目錄結構，在 NTFS 卷中，文件在目錄中以
B+樹的形式排列，在目錄中查找文件時按
B+樹的搜索方法先搜索根節點(從根目錄開始)，然後按要找的文件名與根節點中的子節點對應的文件名相比較以確定在哪個子節點對應的存儲區中搜索，然後以子節點為當前的根節點再搜索，直到找到文件為止。 ^[1]^

微軟系統提供的搜索雖然可以搜索文本內容，但速度十分不理想。大多數情況下，我們只想知道文件存放在電腦的哪個文件夾下，而本程序很好的解決了這一點。

2 NTFS 簡要介紹

2.1 NTFS

NTFS(新技術文件系統)是Windows
NT 以及之後的 Windows 的標准文件系統。 NTFS 取代了文件分配表(FAT)文件系統，為 Microsoft 的 Windows 系列操作系統提供文件系統。 NTFS 對 FAT 和 HPFS(高性能文件系統)作了若干改進，例如，支持元數據，並且使用了高級數據結構，以便於改善性能、可靠性和磁盤空間利用率。 ^[2]^

2.2 現狀

隨著以 NT 為內核的 Windows
2000/XP 的普及，很多個人用戶開始用到了 NTFS。 NTFS 也是以簇為單位來存儲數據文件，但 NTFS 中簇的大小並不依賴於磁盤或分區大小。簇尺寸縮小不但降低了磁盤空間的浪費，還減少了產生磁盤碎片的可能。 NTFS 支持文件加密管理功能，可為用戶提供更高層次的安全保證。 ^[3]^

3 準備

3.1 設計思路

讀取磁盤全部文件名稱作為數據，根據用戶輸入關鍵字，在數據中匹配。正確的連同路徑一起返回給用戶，以方便的打開文件。

具體設計思路如圖 3-1 所示。

哈希表最大的優點，就是把數據的存儲和查找消耗的時間大大降低，幾乎可以看成是常數時間；而代價僅僅是消耗比較多的內存。然而在當前可利用內存越來越多的情況下，用空間換時間的做法是值得的。另外，編碼比較容易也是它的特點之一。

3.2 什麼是 USN

程序的重中之重是讀取 USN。

USN是更新服務編號日誌或更改
Journal 的縮寫，對 NTFS 卷裡所修改過的信息進行相關記錄的功能，可以在分區中設置監視更改的文件和目錄的數量，記錄下監視對象修改時間和修改內容。

當啟用 USN 日誌時，對於每一個 NTFS 卷，當有添加、刪除和修改文件的信息時，NTFS 都使用 USN 日誌記錄下來，並儲存為
USN_RECORD 的格式。

3.3 為什麼快速

USN 日誌相當於 WORD 目錄，提供了索引，當然文章內容髮生變化的時候，USN 日誌會記錄下來何時做了修改，但它並不記錄裡面具體修改了什麼東西，所以索引文件很小。而當你想查找文章具體的段落時，你就不用狂轉鼠標滾輪，直接看目錄即可，定位也只需要按住 ctrl+鼠標單擊。

同理，當你想查找某一篇文件時，可以直接通過查找 USN 日誌(也就是建立的索引)就知道這個文件是否存在。

PS：windows 雖然不是“一切皆是文件”(Unix/Linux
的基本哲學之一)，但文件夾也是以文件的形式存在，所以也可以通過 USN 來查找位置。

4.2 構建查找數據

4.2.1 構建 Vector

向量：

一種順序容器。 vector 中的元素通過其位置下標訪問。可通過調用 push_back 或
insert 函數在 vector 中添加元素。在 vector
中添加元素可能會導致重新為容器分配內存空間，也可能會使所有的迭代器失效。在
向量
容器中間添加(或刪除)元素將使所有指向插入(或刪除)點後面的元素的迭代器失效。

用Vector 來存放<文件名，當前目錄>，由於模糊查找的要求，二分查找之類的快速查找方式，在這裡反而不適用，而Vector 從begin 到end 的線性遍歷方式，反而比較符合這一要求。

4.2.2 構建哈希表

這裡利用 STL 提供的 map 庫函數^[5]^，map 是鍵－值對的集合。 map
類型通常可理解為關聯數組(associative
array)：可使用鍵作為下標來獲取一個值，正如內置數組類型一樣。而關聯的本質在於元素的值與某個特定的鍵相關聯，而並非通過元素在數組中的位置來獲取。

  1. 考慮到若是自己實現哈希函數，可能有若干 bug，影響進度，不如使用 STL 提供的穩定的 map 來實現
  2. “Don’t Reinvent the Wheel”不要重複發明輪子^[6]^

4.4 加入線程

4.4.1 為什麼加入線程

線程的好處：線程價廉。線程啟動比較快，退出比較快，對系統資源的衝擊也比較小。

另外考慮到：

  1. 由於 io 的限制，對同一塊硬盤，用多線程同時讀取 MFT，意義不是很大
  2. 磁盤訪問位於同一硬盤不同分區的 MFT，有可能反而會影響速度。而對於多塊硬盤的電腦，應該可大大提高效率。但是多塊硬盤的用戶，可能組成了 RAID 磁盤列陣。最好的解決方案應該就是 A-Z 盤順序讀取。
  3. 最初創建 MFC 時，順序執行，結果導致界面在全部數據統計結束後才能顯示，在此過程中，用戶很容易失去耐心。
  4. 最終決定一個 UI 線程，一個 worker 線程後台統計數據。

4.4.2 在 MFC 中啟動一個 Worker 線程

MFC 早加入了對多線程的支持。在一個典型的 MFC

用 AfxBeginThread 來啟動線程。

4.5 查找

4.5.1 通配符

實際應用中，文件名不可能清楚的記得，比如 test.2012-5-14.txt，經常是輸入“test
.txt”，所以需要模糊查找。由於在 console 下，以<filename,
pfrn>創建哈希表，如果要實現模糊查找，哈希表變沒有必要，可以直接利用容器 vector。

根據實際情況，不需要很嚴格的通配符，”* ？”
即可解決大部分查找問題，加上用戶一般可能不知道通配符的使用，更不用說正則表達式。所以以“空格”
