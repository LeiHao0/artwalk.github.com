소화 할 수없는 프로그래밍 언어는 나쁜 언어입니다

작성일 2014-12-23 Edited on 2020-12-21 Disqus:
Symbols count in article: 1.1k Reading time ≈ 1 mins.

갭 이어 때이 책을 봤어요. codecademy에 질문을 쓰고 있었어요. 루비 챕터 만 봤어요. 그 당시에는 ‘Go’가 이미 괴짜라는 걸 알았습니다.

1 년 후 제가 일하는 회사가 백엔드 구조 조정을 할 때 ‘스칼라’에 대해 생각했습니다. 친구 중 한 명이 경기를보고 고임금을 냈기 때문에 단호하게 ‘얼랑’을 보러 갔어요. 클로저`

이 책을 다시 읽었을 때 ‘Ruby, Io, Prolog, Scala, Erlang, Clojure, Haskell’이

우물가에 앉아 하늘을 다시 보니이 책을 다시 읽다가 읽을 필요가 없다는 것을 알게되었습니다

그러니 호기심이 많고 호기심이 만족 스러우면 한번 보시고 진지하게 공부하고 싶다면 건너 뛰세요.

프로그래밍 언어를 살펴보면 다음 범주에서 벗어날 수 없을 정도로 너무 많습니다.

  1. 모델
  • 기능의
  • OO
  • 원기
  1. 데이터 구조 알고리즘
  2. 데이터 구조
    • 유형
    • 연결된 목록
    • 해시 테이블
  3. 방법
    • 심판
    • 주기
  4. 병발 사정

동시성에 관해서는 프로세스가 인간의 사고에 부합하지 않기 때문에 현재로서는 좋은 해결책이없는 것 같습니다.
미래에 해결 되더라도 실력있는 사람들이 써서 캡슐화해야한다고 생각합니다. 그래서 평범한 IQ를 가진 사람들이 ‘ifelse’를 쓰는 것처럼 간단하게 풀 수 있도록

개인적으로 언어의 인기는 다음과 같은 점에서 분리 할 수 ​​없습니다.

  1. 상업용 프로모션 (Java | OC)
  2. 안전하고 안정적인
  3. 인간이 읽기 쉬운가?

처음 두 포인트는 말할 필요 없어 너무 많이 말하면 눈물이 될거야

7주의 7 개 언어로 언급됩니다.
여러면에서 Clojure는 현명한 쿵푸 마스터, 언덕 위의 신탁 또는 수수께끼의 제다이 트레이너입니다.

Lisp의 이점은 “The Hacker and the Painter”에서 알려졌으며 Emace는 “신의 편집자”로 칭송 받았습니다.

하지만 너무 좋고 너무 멋지지만 인기가 없어서 생각할 가치가있는 질문

통사론 적 설탕의 관점에서 나는 이전 세대 (C / C ++ / Java) 프로그래머 (non-derogatory)였던Swift를 한 번 사용했던 것을 기억합니다. 방금 반환 값이->로 변경된 것을 보았습니다. 변수 유형은 다음과 같습니다. 뒤에서 외쳤다
나 자신은 ‘고 스위프트’와 다른 문법적 당분을 완전히 받아들이지 만 ‘리스프’의 접두사 표현이 모두 불편한 느낌이 든다. 단순함이 좋고 길면 어지럽고 닫는 괄호가 매우 이단 적이다.
프로그래머가 시스템에 익숙해 진 후 새로운 구문을 받아들이는 것이 얼마나 어려운지 알 수 있습니다.

“코드는 사람들이 볼 수 있도록 작성되었습니다”

그래서 저는 반 인간적 독서 인 언어가 사라지거나 적어도 인간에게 인기가 없을 것이라고 생각합니다.

그래서 Mazi는 프로그래머가 코드를 작성하는 것을 기쁘게 만드는 방법에 초점을 맞춘 다른 접근 방식을 취했습니다.

그래서 최고의 언어는없고 가장 적합한 언어 만

물론 컴퓨터가 코드를 작성할 수 있도록 진화 한 상황에서는 위의 어느 것도 사용되지 않습니다.

이것은 인간의 패러다임에서 벗어 났기 때문에
